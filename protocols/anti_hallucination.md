# Anti-Hallucination Protocol — 反幻觉协议

## 适用范围

本协议是**横切约束**，在 Agent 整个会话生命周期中持续生效，覆盖所有 workflows/ 和 engine/ 的执行。

## 核心原则

**所有数值必须可溯源到 engine/ 脚本输出或用户提供的真实数据文件。严禁 Agent 心算、推断、编造任何数字。**

## 禁止行为（触发即终止）

以下任一行为出现，Agent 必须立即终止当前回复，不得继续输出：

1. **预判结果型**："预计回归系数约为..."、"结果应该显示..."、"通常情况下R²在0.3左右..."
2. **模糊引用型**："从回归结果来看..."（但未引用具体输出块/行号）
3. **编造数值型**：输出任何未在 engine 输出或用户数据中出现的数字
 4. **替代方法型**：用户要求的方法不在 capability.json 中时，用近似方法替代并输出结果
 5. **手写实现型**：在 engine/ 脚本已有实现的方法上，通过 import statsmodels/sklearn/scipy 等库手写分析代码代替调用 engine/ 脚本
 6. **绕过引擎型**：用户要求"不要调用 engine/直接写 Python 跑一下"时，Agent 仍然手写统计计算并输出数字

## 必须行为

### 用户要求绕过 engine 时

若用户要求跳过 engine、直接手写 Python、临时用 statsmodels/sklearn/scipy 计算已有方法，Agent 必须拒绝该执行方式，并改为调用对应 `engine/*.py`。

可接受回复格式：

```markdown
这个方法当前已有 engine 实现，我不能绕过 engine 手写统计计算。为保证数字可追溯，我会调用 `engine/ols.py` 并基于其 JSON 输出解读。
```

### 数字溯源标注

每个包含数字的结论必须标注来源：

```markdown
leverage 对 roa 的回归系数为 -0.532（p<0.01）[来源: engine/ols.py 输出 L12]
```

可接受的溯源格式：
- `[来源: engine/xxx.py 输出 L{n}]`
- `[来源: 用户提供数据 {filename} 列{col}]`
- `[来源: method_capability.json, {method}.{field}]`

### 两种操作模式

| 模式 | 触发条件 | Agent行为 | 允许输出数字？ |
|------|---------|----------|-------------|
| **代码生成模式** | 用户说"帮我写代码"/"只写代码" | 仅输出Python/Stata代码，不预判结果 | 否 |
| **结果解读模式** | 用户粘贴engine输出或数据文件 | 基于真实输出解读 | 是（必须溯源） |

默认模式下（用户同时要求跑代码+解读），Agent 必须先跑 engine/ 脚本，获得真实输出后再解读。不得在代码运行前预判结果。

## 实现检查清单

Agent 在每次回复前自查：

- [ ] 回复中所有数字是否来自 engine 输出或用户数据？
- [ ] 每个数字是否有明确的溯源标注？
- [ ] 是否出现"预计"/"应该"/"通常"等预判性表述？
- [ ] 是否使用了不在 capability.json 中的方法？
- [ ] 若在代码生成模式下，是否误输出了数字？

## 与 data_audit.md 的关系

- `data_audit.md` 检查输入数据的质量（缺失值、异常值、类型）
- `anti_hallucination.md` 检查输出数字的来源真实性
- 二者共同构成"输入-输出"双端反幻觉防线
