# Data Audit Protocol — 数据质检协议

## 位置

前置阻断 [P1]：在 protocol_order.md 中位于 method_selection 之前。
数据不通过质检 → 拒绝执行后续 workflow → 返回质检报告。

## 质检维度

### 1. 文件级检查

| 检查项 | 通过条件 | 不通过处理 |
|--------|---------|-----------|
| 文件可读 | 格式为 .csv/.xlsx/.parquet/.dta | 提示支持的格式列表 |
| 编码正确 | UTF-8 / GBK 可正常解码 | 尝试自动检测编码 |
| 非空文件 | 行数 ≥ 2（含表头至少1行数据） | 拒绝，提示提供有效数据 |

### 2. 结构级检查

| 检查项 | 通过条件 | 不通过处理 |
|--------|---------|-----------|
| 列名有效 | 无空列名、无重复列名 | 自动修复空列名；列出重复列名让用户选择 |
| 数值列存在 | 至少1列数值型变量 | 拒绝，提示数据中无可分析的数值列 |
| 样本量足够 | N ≥ 30（回归方法N ≥ 50） | 警告但继续（标注样本量风险） |

### 3. 数据质量检查

| 检查项 | 阈值 | 不通过处理 |
|--------|------|-----------|
| 缺失率 | 单列缺失率 > 50% | 建议删除该列 |
| 缺失率 | 单列缺失率 20-50% | 警告，建议处理 |
| 异常值 | IQR × 3 之外的值 | 列出，询问是否缩尾处理 |
| 常数列 | 标准差 = 0 | 建议删除（回归中无信息量） |
| 类别过多 | 分类变量类别数 > 样本量的10% | 警告（可能不适合做分组变量） |

### 4. 方法匹配检查

根据用户选择的方法，检查数据是否满足前置条件：

| 方法 | 必须满足 | 检查方式 |
|------|---------|---------|
| panel | 存在 entity_col 和 time_col，且组合唯一 | 检查 diagnose.py 输出 |
| did | 存在二值处理变量 + 前/后时间期 | 检查处理变量取值 |
| psm | 处理变量为二值 + 有足够协变量 | 检查处理变量分布 |
| mediation | 中介变量为连续数值型 | 检查 mediator 列类型 |
| moderation | 调节变量为数值型或二值分类 | 检查 moderator 列类型 |
| heterogeneity | 分组变量类别数 2-6 | 检查 group_var 取值 |

## 阻断决策

质检结果分三级：

- **PASS**：全部通过 → 继续到 method_selection
- **WARN**：存在警告项（如缺失率20-50%、异常值）→ 告知用户风险，用户确认后继续
- **BLOCK**：存在阻断项（如文件不可读、无数值列、方法前置条件不满足）→ 终止，返回质检报告

## 质检报告格式

```markdown
## 数据质检报告

**文件**：{filename}
**样本量**：{N} 行 × {M} 列
**总体评估**：PASS / WARN / BLOCK

### 阻断项（如有）
- {问题}：{建议}

### 警告项（如有）
- {问题}：{建议}

### 方法匹配
- 用户请求方法：{method}
- 数据是否满足前置条件：是 / 否（{具体问题}）
```

## 例外

用户可以通过明确指令"跳过质检"绕过本协议，但 Agent 必须在最终输出中标注：
> ⚠️ 用户选择跳过数据质检，以下结果未经过数据质量验证。
